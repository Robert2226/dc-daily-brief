On a project plan, that shows up as a shorter risk register and a compressed schedule for the "design" phase, because you're executing a known pattern instead of doing discovery. Recognizing when you're buying a reference architecture vs. a custom build is the single fastest way to sanity-check a vendor's proposed timeline.

## Networking
- **Cisco Live Protect ends the patch-and-pray cycle for Nexus** — Cisco detailed Live Protect's real-time threat mitigation for Nexus switching infrastructure, shielding known vulnerabilities the moment they're disclosed instead of waiting for a maintenance window to apply a patch. [Network World](https://www.networkworld.com/article/4191767/cisco-details-live-protects-real-time-threat-mitigation-capabilities.html)
- **Another Cisco SD-WAN zero-day, exploited before the fix shipped** — SecurityWeek reports attackers exploited a Cisco SD-WAN zero-day for months before a patch was available, the latest in a steady drumbeat of Cisco vulnerability disclosures this year. [SecurityWeek](https://www.securityweek.com/cisco-patches-another-sd-wan-zero-day-exploited-in-attacks/)
> **In Practice · Virtual patching vs. the maintenance window.** Traditional patching forces a tradeoff between security (patch now, risk an outage) and availability (wait for a change window, stay exposed). Live Protect-style "virtual patching" — shielding the vulnerable code path at the network layer without touching the device's running config — breaks that tradeoff, which is exactly why it matters for critical-power/DCIM environments where an unplanned reboot has real operational cost. It's the network-security version of a redundant power path: you fix the fault without dropping the load.

[CNCF](https://www.cncf.io/projects/)
- **Accenture breach exposes source code and access keys** — A threat actor claims roughly 35GB stolen from Accenture in an early-July intrusion, including source code and access keys that could let attackers map internal application logic and move through cloud storage and repos. [Cybersecurity Dive](https://www.cybersecuritydive.com/news/accenture-data-breach-access-keys-source-code/824694/)
> **In Practice · GPU fragmentation is a bin-packing problem.** When workloads request whole GPUs but only use a fraction of one, the scheduler ends up with dozens of partially-used cards it can't reassign to new jobs — capacity that's technically free but effectively stranded. It's the compute equivalent of a rack that's fully powered but half-empty on U-space: the constraint isn't total capacity, it's how finely you can subdivide and reallocate it. HAMi and tools like it exist to make GPUs shareable at sub-card granularity, the same way virtualization made a whole server shareable at sub-server granularity twenty years ago.

> **In Practice · PMBOK 8's six principles, and why they replaced twelve.** PMBOK 7 gave you 12 principles to hold in your head; PMBOK 8 collapses them to six more actionable guidelines — the intent is fewer, sharper heuristics you can actually apply mid-decision instead of a checklist you consult afterward. Pair that with the new "case set" question format (a shared scenario with multiple linked questions, versus one isolated prompt) and the exam is testing whether you can carry context across a sequence of decisions the way you would on a real, evolving project — not whether you can recall an isolated fact. For prep starting today: drill case-set-style practice questions specifically, since they're new and under-represented in older question banks.
