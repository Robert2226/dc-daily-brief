When evaluating a growth story, always ask whether the headline number is booked, billing, or delivered.

## Networking
- **Cisco Live Protect uses eBPF to patch vulnerabilities without downtime** — Live Protect deploys kernel-level runtime shields — built on extended Berkeley Packet Filter (eBPF) — that block exploitation of a disclosed vulnerability within minutes, no reboot or maintenance window required, buying time for a permanent patch through normal change-control. Currently available for Nexus 9000 switches via Nexus Dashboard, with campus/branch support planned later in 2026. [Network World](https://www.networkworld.com/article/4191767/cisco-details-live-protects-real-time-threat-mitigation-capabilities.html)
- **Cisco deploys AI-ready network for The Open Championship at Royal Birkdale** — Under a new multiyear partnership with The R&A, Cisco fielded nearly 1,000 wireless access points plus its full routing/switching/security stack as the exclusive network backbone for The 154th Open — a large-scale, time-boxed proving ground for the same secure-network architecture Cisco is selling into enterprise AI deployments. [Cisco Newsroom](https://newsroom.cisco.com/c/r/newsroom/en/us/a/y2026/m07/the-r-a-partners-with-cisco-to-deliver-critical-networking-infrastructure-to-major-championships-and-new-global-headquarters.html)
> **In Practice · eBPF is why "patch without downtime" is now technically credible.** Traditional patching replaces vulnerable code, which usually means restarting the affected process or rebooting. eBPF lets you attach small, sandboxed programs directly into the Linux kernel that observe or intercept specific system calls and network events *without modifying or restarting anything* — so a vendor can insert a narrow rule that blocks the exact exploit path (a bad syscall pattern, a malicious packet shape) while the real fix ships on a normal schedule. It's the same kernel-level hooking technology powering modern observability tools (Cilium, Falco) — one mechanism, security and networking use cases both.

- **Michelin replaces two CNIs with Cilium, zero customer-impacting outages** — Michelin consolidated its Kubernetes networking onto Cilium, retiring two separate Container Network Interface plugins in the process and gaining unified network visibility across its clusters — completed July 14 with no customer-facing downtime during migration. [CNCF](https://www.cncf.io/)
> **In Practice · What a CNI actually does, and why running two is a liability.** A Container Network Interface plugin is what assigns pod IP addresses, programs routing rules, and enforces network policy inside a Kubernetes cluster — it's the layer between "my app has a service definition" and "packets actually move." Running two CNIs (common after a merger, multi-team cluster, or unresolved past migration) usually means duplicated policy engines, inconsistent observability, and higher troubleshooting overhead, since a dropped packet could originate in either one. Michelin's zero-downtime consolidation is the payoff of good migration sequencing — moving workloads incrementally behind existing policy rather than a hard cutover.
